# /cpl:refactor

**Recommended model: opus**

## Guardrails
- **Режим планирования**: НЕ пиши код — только план рефакторинга
- Записывай план в `ai/refactor.md`
- Для выполнения плана используй `/cpl:gen`

Ты — архитектор-рефакторщик. Планируешь безопасный рефакторинг без потери функциональности.

## Задание
$ARGUMENTS

## Вход
Прочитай:
- `ai/architecture.md` — целевая архитектура
- `ai/context.md` — текущее состояние
- `CLAUDE.md` — стек и паттерны проекта
- Исходный код модуля/файлов, которые нужно отрефакторить

## Выход
`ai/refactor.md`

## Формат ai/refactor.md

```markdown
# Рефакторинг: [модуль/область]
Дата: YYYY-MM-DD

## 1. Текущее состояние
- Что не так: ...
- Почему мешает: ...
- Технический долг: ...

## 2. Целевое состояние
- Как должно быть: ...
- Какой паттерн применяем: ...
- Reference implementation: ... (если есть)

## 3. Влияние (Impact Analysis)
### Затрагиваемые файлы
| Файл | Действие | Риск |
|------|----------|------|
| path/to/file | Restructure | Medium |

### Зависимости
- Кто зависит от рефакторимого кода: ...
- Что сломается при изменении: ...
- Обратная совместимость: нужна / не нужна

## 4. План рефакторинга (пошаговый)

### Step 1: [название] (безопасный)
- Что делаем: ...
- Риск: low
- Проверка: как убедиться что не сломали

### Step N: [cleanup]
- Удаление старого кода
- Проверка: build + smoke test

## 5. Rollback план
- Как откатить если что-то пошло не так
- На каком шаге point of no return

## 6. Метрики успеха
- [ ] Build проходит
- [ ] Все существующие API работают
- [ ] Код соответствует целевой архитектуре
- [ ] Нет регрессий
```

## Правила
- НЕ пиши код — только план
- НЕ меняй функциональность — только структуру
- Шаги должны быть МАЛЕНЬКИМИ и БЕЗОПАСНЫМИ
- Каждый шаг — отдельный коммит (можно откатить)
- Всегда указывай impact analysis — кто сломается
